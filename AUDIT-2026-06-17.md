## Executive Summary

The site is **operational and monetized** — Stripe checkout returns live URLs, the AI Training Feed correctly 402s paywalled bots, and the noindex crisis from 2026-06-14 is resolved. However, the **live deployment is currently behind the local working tree**: critical fixes for `/llms.txt`, the Cloudflare Worker, and several API/middleware changes exist locally but are not on the production build. This creates immediate regressions in production that the codebase has already fixed.

There are also **serious security exposures** (Cloudflare API key in git, hardcoded Neo4j fallback password, services running as root), **monetization leaks** (a free JSON repair API serves the same data as the paid feed), and **SEO inconsistencies** (non-US models being submitted to Google via sitemap).

## 3. Monetization

### 🔴 CRITICAL — Free `/api/v1/repair` serves paid training feed data

| | |
|---|---|
| **Files** | `src/middleware.ts:116-134`, `src/app/api/v1/repair/route.ts` |
| **Behavior** | Any `/repair/{year}/{make}/{model}/{task}` request with `Accept: application/json` is rewritten to `/api/v1/repair`, which has **zero payment gating** |
| **Impact** | Agents can get the same repair data that `/api/data/...` and `/api/premium-repair-data` charge $0.01/page for |
| **Fix** | Gate `/api/v1/repair` identically to `/api/data/...`, or restrict JSON rewrite to authenticated sessions only. |

### 🔴 CRITICAL — x402 payment verification not forwarded to route handlers

| | |
|---|---|
| **Files** | `src/middleware.ts:275`, `src/lib/paymentGate.ts:47` |
| **Behavior** | Middleware sets `X-Payment-Verified: x402` on the **response** object, but route handlers read it from `request.headers`. Next.js middleware response headers are not automatically request headers. |
| **Impact** | x402-paying agents hit `checkStripeAccess`, fail the Stripe-key check, and get `402 invalid_api_key` even though they already paid |
| **Fix** | Use `NextResponse.next({ request: { headers: requestHeaders } })` to pass the verified flag. |

### 🟡 HIGH — Stripe webhook double-credit risk

| | |
|---|---|
| **File** | `src/app/api/stripe/webhook/route.ts:27-58` |
| **Behavior** | `checkout.session.completed` events are processed without checking whether the session was already credited |
| **Impact** | Retries or duplicate deliveries double/triple customer credits |
| **Fix** | Check transaction history for the session ID before crediting, or make `(stripe_session_id, type)` unique for `stripe_purchase`. |

### 🟡 HIGH — Refund/dispute events not recorded

| | |
|---|---|
| **Files** | `src/app/api/stripe/webhook/route.ts:63-84`, `src/lib/credits.ts:155` |
| **Behavior** | `creditCustomer()` throws when `amountCents <= 0`; refund/dispute events attempt to log `$0` audit transactions |
| **Impact** | Webhook returns 200 to Stripe but no audit row is written; paper trail is lost |
| **Fix** | Add a dedicated zero-amount audit logger or allow `amountCents === 0` for audit-only types. |

### 🟡 HIGH — HEAD requests debit credits

| | |
|---|---|
| **Files** | `src/app/api/data/[year]/[make]/[model]/[[...section]]/route.ts:343-372`, `src/app/api/premium-repair-data/route.ts:62-72` |
| **Behavior** | `HEAD` calls `checkStripeAccess()`, which calls `debitPageRequest()` and subtracts 1 cent |
| **Impact** | HEAD should be a free existence/probe request |
| **Fix** | Skip billing in `HEAD` handlers. |

### 🟡 HIGH — Paid API can charge before returning 404

| | |
|---|---|
| **File** | `src/app/api/data/[year]/[make]/[model]/[[...section]]/route.ts:403-418` |
| **Behavior** | `checkStripeAccess()` runs before validating the requested section/task |
| **Impact** | Customer is charged, then receives 404 for unknown section |
| **Fix** | Validate content exists before debiting; refund or avoid debit on 404. |

### 🟡 MEDIUM — Affiliate links earn $0 commission

|---|---|
| **Files** | `src/services/affiliateService.ts:59-73`, `src/lib/impactAffiliate.ts:27-69` |
